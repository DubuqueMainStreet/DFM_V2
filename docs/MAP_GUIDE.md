## Architecture

All UI controls (date selector, search, filters, legend) live inside the iframe. The Velo page only fetches data and communicates via `postMessage`.

```
MAP.mggqp.js (Velo)          vendor-map-full-ui.html (Leaflet)
  │                              │
  ├─ Queries CMS collections     ├─ Date selector
  ├─ Handles geolocation         ├─ Search box
  ├─ Sends data via postMessage  ├─ Category filters
  └─ Test data fallback          ├─ Vendor markers + popups
                                 ├─ Stall polygons + POIs
                                 └─ Legend, locate, fit market
```

### postMessage Protocol

| Direction | Type | Payload |
|-----------|------|---------|
| iframe → Velo | `iframeReady` | — |
| iframe → Velo | `requestDateData` | `{ date: "YYYY-MM-DD" }` |
| iframe → Velo | `requestLocation` | — |
| Velo → iframe | `setMarketDates` | `{ dates: [{ value, label }] }` |
| Velo → iframe | `loadMapData` | `{ vendorsOnDate, allStallLayouts, allPois, currentDate, isTestData? }` |
| Velo → iframe | `mapLoading` | — |
| Velo → iframe | `locationResult` | `{ latitude, longitude, accuracy }` |
| Velo → iframe | `locationError` | `{ message }` |

## Test Data Mode

`MAP.mggqp.js`: `const USE_TEST_DATA_DEFAULT = false;` (production).

Force test mode on-demand with `?testData=1` in the URL (works regardless of the default). Use for developer QA against `scripts/output/*REAL.geojson`-derived fixtures.

Test data is generated by `scripts/generateMarketTestData.js` from real GeoJSON/CSV files in `scripts/output/`.

### Pre-flip checklist — verify CMS is populated

Before setting `USE_TEST_DATA_DEFAULT = false`, confirm the four map
collections contain real 2026 data. Open the **dataParse** admin page
and, in the browser console, run:

```javascript
await window.checkMapData()
```

This calls `checkMapDataCollections()` from
`src/backend/diagnosticCheck.jsw` and logs a table like:

| collection        | count | hasRequiredFields |
|-------------------|-------|-------------------|
| MarketDates2026   | 27    | true              |
| StallLayouts      | 60+   | true              |
| POIs              | 5+    | true              |
| MarketAttendance  | 100+  | true              |

Minimums expected:
- `MarketDates2026`: ≥ 20 rows (every 2026 market Saturday) with `title` and `date`.
- `StallLayouts`: ≥ 10 rows with `stallId`.
- `POIs`: ≥ 1 row (entrance, token booth, restrooms, etc.).
- `MarketAttendance`: ≥ 1 row with a valid `dateRef`.

If any collection is empty or missing required fields, run the matching
import on the dataParse page:
- Vendors → `importVendorRoster`
- Stalls + POIs → `importGeoJsonFeatures`
- Attendance → `importStallAssignments`

Only once `result.ready === true` should you flip
`USE_TEST_DATA_DEFAULT` to `false` and publish.

## Geolocation Workaround

Browsers block `navigator.geolocation` inside cross-origin iframes. The parent (Wix) requests location using `wixWindowFrontend.getCurrentGeolocation()`, then posts coordinates to the iframe.

1. User taps "Locate me" in iframe
2. iframe sends `requestLocation` to parent
3. Velo calls `wixWindowFrontend.getCurrentGeolocation()`
4. Velo posts `locationResult` (or `locationError`) back to iframe
5. iframe updates marker and optional accuracy circle

Accuracy circle is capped at 45m display radius to avoid dominating the market area.

## Performance Optimizations

- **Static data cache:** Stall layouts and POIs cached after first fetch, reused across date changes
- **CSS containment:** `contain: layout style paint` on body and map container
- **Deferred geolocation:** Uses `requestIdleCallback` to avoid blocking initial render
- **postMessage batching:** 16ms window for style updates (aligns with 60fps), critical updates process immediately
- **Resource hints:** `preconnect` for CDNs, `dns-prefetch` for tile domain
- **Font swap:** `display=swap` for faster text rendering
- **Progressive loading:** Map renders while data loads in background

### Mobile (70%+ of traffic via QR codes)
- `preferCanvas: true` for better mobile rendering
- Large tap targets on all controls
- Touch event handling for map interaction
- Throttled localStorage writes (500ms)

## Map Features

- **Date selector** — Pick market date, loads vendor data for that date
- **Category filters** — Ready to Eat, Farm Fresh, Baked Goods, Coffee, SNAP/EBT, Info, Restrooms, Seating, Parking, Tokens
- **Search** — Filter vendors by name or description (300ms debounce)
- **Vendor markers** — Color-coded by type (100+ keyword rules), custom divIcons
- **Stall polygons** — Color-coded by zone, tooltips with stall IDs
- **POI markers** — Info, restrooms, seating, parking with icons
- **Popups** — Vendor details with directions link, Prev/Next navigation
- **Focus mode** — Non-focused markers dim when a popup is open
- **Fit market** — One-tap button to fit map to market bounds
- **Legend** — Toggleable vendor type legend
- **Reduced motion** — Respects `prefers-reduced-motion: reduce`
- **DFM branding** — Custom fonts (TAY Milkbar, Poppins, Outdoors Inks), brand colors

## Known Limitations

1. **Geolocation in iframe** — Browser Permissions Policy blocks `navigator.geolocation` in cross-origin iframes. Handled via parent-posted `locationResult` (see Geolocation Workaround above).
2. **Map tile CDN dependency** — Tiles load from CartoDB/OSM over HTTPS. No offline fallback; a network failure will leave the map blank apart from vendor markers.
3. **Query result caps** — CMS queries use `.limit(1000)` for stalls, POIs, dates, and attendance. Close to the ~1,350 expected assignment rows for the 2026 season; switch to pagination if the dataset grows.
4. **Browser support** — ES6+ modern browsers only. IE11 is not supported.

## Wix Editor Deployment

The Wix-side setup is a one-time configuration of the HTML embed that hosts the iframe.

### Required elements on the MAP page

1. **HTML component** with ID `#mapFrame`
   - Source: `/vendor-map-full-ui.html` (from the site `public/` folder)
   - Size: 100% width and height, margins 0

2. **No legacy UI elements** — the following should NOT be on the page (all UI now lives inside the iframe):
   - `#datepicker1` (date picker)
   - `#searchInput` (search input)
   - `#clearAllButton` (clear filters button)
   - Per-category filter buttons (`#readyToEatButton`, `#farmFreshProduceButton`, `#bakedGoodsSweetsButton`, etc.)

After editing the HTML component, publish via `npx wix publish` or the Wix Editor.

### Customizing filters

Filter buttons are declared in `vendor-map-full-ui.html` with three possible matching rules via `data-type`:

| `data-type` | Description | Example `data-value` |
|-------------|-------------|----------------------|
| `vendorType` | Match vendor type exactly | `"Grower/Producer/Processor"` |
| `keyword` | Search space-separated keywords against name/description | `"coffee espresso latte tea"` |
| `poiType` | Match POI type | `"Restroom"` |

## Troubleshooting

**Map not loading**
1. Confirm `#mapFrame` exists and its Source is `/vendor-map-full-ui.html`.
2. Open the browser console for iframe errors (look for CORS or tile failures).
3. Check Network tab for 404 on the HTML file.

**Dates not populating**
1. Verify `MarketDates2026` has rows with `title` populated.
2. Check the Velo site console for query errors.
3. Confirm permissions allow anonymous `wix-data` reads on the collection.

**Vendors not showing for a date**
1. Verify `MarketAttendance` has records whose `marketDate` matches the selected date.
2. Confirm `vendorRef` references resolve and `stallId` matches a `StallLayouts` row.
3. If the map shows "0 vendors", toggle `?testData=1` in the URL — if it renders, the issue is data, not code.
